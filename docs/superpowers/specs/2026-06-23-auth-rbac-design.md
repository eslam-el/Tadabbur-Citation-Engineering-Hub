# وثيقة تصميم: المصادقة ونظام الأدوار والصلاحيات (Auth + RBAC)

**التاريخ:** 2026-06-23
**الحالة:** معتمدة من المستخدم — جاهزة لكتابة خطة التنفيذ.

## الهدف
تحويل المنصة من «وصول مفتوح بلا هوية» إلى نظام **دفاع في العمق** من جدارين:
1. **المصادقة (Authentication):** لا أحد يصل للبيانات قبل تسجيل دخول موثوق عبر Google.
2. **الصلاحيات والأدوار (Authorization/RBAC):** المدير وحده يدير الأعضاء؛ المستخدمون يضيفون البلاغات ويصدّرونها فقط.

بالإضافة إلى أداة **حذف البيانات التجريبية** لمنع اختلاطها بالبيانات الحقيقية.

---

## القرارات المعتمدة
- **آلية الدخول:** Google SSO (تسجيل ذاتي مفتوح عبر الرابط) — لا قائمة بيضاء مغلقة.
- **بوّابة الموافقة (الجدار الثاني):** كل مسجَّل جديد يبدأ بحالة `PENDING` ولا يرى أي بيانات حتى يعتمده مدير.
- **المدير الجذري (Root Admin):** يُحدَّد بمتغيّر البيئة `ADMIN_EMAIL`، ويملك صلاحية `ADMIN` و`ACTIVE` دائماً، ولا يُخفَّض ولا يُحذف.
- **مكتبة المصادقة:** الترقية إلى **Auth.js v5** (`next-auth@5`) — مصمّمة لـ App Router في Next 16 (دالة `auth()` + middleware).
- **فصل الصلاحية عن المسمّى الوظيفي:** حقل `role` النصّي يبقى مسمّىً وظيفياً (مراجع/محقق…)؛ وتُضاف صلاحية مستقلّة في `accessLevel`.
- **الهوية = العضو:** الجلسة المصادَقة تحدّد العضو الحالي تلقائياً (إلغاء حيلة `localStorage`).

---

## المعمارية العامة
- **استراتيجية الجلسة:** JWT (لا جداول جلسات في قاعدة البيانات). الحمولة (token/session) تحمل: `memberId`, `email`, `name`, `accessLevel`, `status`.
- **ثلاث طبقات فرض:**
  1. **الصفحات** عبر `middleware.ts`: غير المصادَق → `/login`؛ المصادَق `PENDING` → `/pending`؛ `ACTIVE` → التطبيق.
  2. **الـ API** خادمياً: كل واجهة تتحقق من الجلسة والصلاحية قبل التنفيذ (الإخفاء في الواجهة وحده لا يكفي).
  3. **الواجهة**: إخفاء أزرار الإدارة عن غير المديرين (تحسين تجربة فقط).

## نموذج البيانات — تعديل `Member`
تُضاف الحقول التالية (هجرة Prisma، دون المساس ببنية البلاغات):

```prisma
model Member {
  // ... الحقول الحالية تبقى كما هي (name, role, color, initials, active, createdAt, updatedAt)
  email        String?  @unique            // بريد جوجل — فريد. NULL = حساب غير حقيقي (تجريبي/قديم)
  accessLevel  String   @default("USER")    // "USER" | "ADMIN"  ← الصلاحية
  status       String   @default("PENDING") // "PENDING" | "ACTIVE"  ← بوابة الموافقة
  image        String?                      // صورة جوجل (اختياري)
  seeded       Boolean  @default(false)     // true لسجلات البذر — احتياط للتمييز المستقبلي
}
```

**ملاحظات النموذج:**
- `role` (الحالي) **لا يتغيّر معناه** — يبقى مسمّىً وظيفياً نصّياً.
- الأعضاء التجريبيون القدامى يبقون كسجلّات تاريخية (مؤلّفو بلاغات سابقة) حتى يُحذفوا عبر أداة التنظيف. `email` لديهم = `NULL`.
- حقل `active` الحالي يبقى كما هو (لم يُستخدم للتحكم بالدخول)؛ التحكم بالدخول عبر `status`.

## منطق الدخول والبوّابة (Data Flow)

```
دخول بجوجل (callback)
  ├─ البريد == ADMIN_EMAIL ؟ → upsert: accessLevel=ADMIN, status=ACTIVE
  ├─ سجل Member موجود بهذا البريد ؟ → استخدمه كما هو
  └─ جديد ؟ → أنشئ Member { email, name(من جوجل), image, accessLevel:"USER", status:"PENDING", seeded:false }
        ↓
توجيه حسب status:
  ACTIVE  → التطبيق كاملاً (القدرات حسب accessLevel)
  PENDING → /pending : «حسابك بانتظار اعتماد المدير» (+ إمكانية ضبط اسمه)
```

- يُنفَّذ الـ upsert في رد نداء (callback) داخل `lib/auth.ts` (`signIn`/`jwt`).
- المدير الجذري يُفرض برمجياً في كل مرة (حتى لو عُدِّل سجله يدوياً).

## قواعد الصلاحيات (تُفرض خادمياً في كل واجهة)

| العملية | PENDING | USER (ACTIVE) | ADMIN (ACTIVE) |
|---|---|---|---|
| رؤية البلاغات/الإحصائيات/النشاط | ❌ | ✅ | ✅ |
| إضافة/تعديل بلاغ، حلول، تعليقات | ❌ | ✅ | ✅ |
| تصدير Excel | ❌ | ✅ | ✅ |
| اعتماد عضو (`PENDING`→`ACTIVE`) | ❌ | ❌ | ✅ |
| ترقية/خفض (`USER`↔`ADMIN`) | ❌ | ❌ | ✅ |
| حذف/تعديل بيانات عضو آخر | ❌ | ❌ | ✅ |
| تعديل ملفه الشخصي (اسمه/لونه) | ❌ | ✅ | ✅ |
| بذر بيانات تجريبية / حذفها | ❌ | ❌ | ✅ |

**حمايات إلزامية:**
- المدير الجذري (`ADMIN_EMAIL`) لا يُخفَّض ولا يُحذف (ضد القفل الذاتي).
- لا يستطيع المدير خفض/حذف نفسه إن كان آخر مدير (تحقق إضافي اختياري — يكفي حماية الجذري).

## أداة «حذف البيانات التجريبية»

**المشكلة المكتشفة:** علاقة `ErrorReport.author` إلزامية بلا `onDelete`، فحذف عضو له بلاغات **يفشل** بخطأ قيد مفتاح أجنبي. لذا زر الحذف الحالي معطوب للأعضاء الوهميين (ولهم بلاغات).

**التمييز:** الوهمي/التجريبي = `Member` بحالة `email = NULL` (الحسابات الحقيقية تملك بريد جوجل دائماً). علم `seeded` يُستخدم احتياطاً.

**العملية (في `db.$transaction` واحدة، بالترتيب الآمن):**
1. حدّد الأعضاء المستهدفين: `email = NULL`.
2. صفّر `solutionAuthorId` في أي بلاغ يشير إلى مستهدَف لكن مؤلّفه **ليس** مستهدَفاً (حماية بيانات حقيقية).
3. احذف `ErrorReport` حيث `authorId ∈ المستهدفين` (تعليقاتها تُحذف تتاليّاً عبر `onDelete: Cascade` الموجود على `ReportComment`).
4. احذف `ActivityLog` حيث `actorId ∈ المستهدفين`.
5. احذف الأعضاء المستهدفين.

**الواجهة:** زر «حذف البيانات التجريبية» في لوحة إدارة الفريق (مدير فقط)، بتأكيد صريح يذكر عدد الأعضاء/البلاغات التي ستُحذف. يُرجِع العدد المحذوف.

**إصلاح الحذف العام:** `DELETE /api/members/[id]` يُحدَّث ليحذف بأمان (تصفير المراجع/حذف البلاغات التابعة ضمن معاملة) بدل الفشل الصامت — مع رسالة واجهة صادقة.

## التعامل مع الأخطاء وحالات الحافّة
- **`ADMIN_EMAIL` غير مضبوط:** تحذير في السجلّ؛ لا مدير جذري → لا يمكن اعتماد أحد. يُوثَّق كمتطلب إلزامي قبل النشر.
- **بريد جوجل بلا اسم:** يُملأ الاسم من حقل جوجل، ويُتاح تعديله في شاشة `/pending`.
- **محاولة USER استدعاء واجهة إدارية مباشرة:** يردّ الخادم `403`.
- **محاولة PENDING الوصول لأي بيانات:** يردّ الخادم `403` وتوجّه الصفحات إلى `/pending`.
- **تكرار محتمل:** لو سجّل عضو حقيقي بنفس اسم عضو تجريبي قديم، يُنشأ سجل جديد منفصل (مرتبط بالبريد)؛ يُنظَّف القديم عبر أداة حذف البيانات التجريبية.
- **حذف عضو له بلاغات:** يُعالَج بمعاملة آمنة (لا فشل صامت).

## التحقق
- **اختبار وحدة** لدوال الصلاحية النقية في `lib/auth.ts` (`isAdmin`, `isActive`) — منطق خالص قابل للاختبار دون شبكة/قاعدة بيانات.
- **بناء** `next build --webpack` بنجاح و0 أخطاء TypeScript.
- **قائمة تحقق يدوية حيّة** (محلياً ثم على Vercel):
  1. دخول حساب جديد → `PENDING` → شاشة الانتظار، لا بيانات.
  2. دخول المدير الجذري → `ADMIN`+`ACTIVE` مباشرة.
  3. المدير يعتمد العضو الجديد → يصبح `ACTIVE` ويرى البيانات.
  4. المدير يرقّي عضواً → يظهر له أزرار الإدارة؛ ثم يخفضه → تختفي.
  5. `USER` لا يرى أزرار الإدارة؛ واستدعاء واجهة إدارية مباشرة يردّ `403`.
  6. «حذف البيانات التجريبية» يحذف الأعضاء بلا بريد وبلاغاتهم فقط، ويُبقي الحقيقية.
  7. تسجيل الخروج يعيد إلى `/login`.
  8. المدير الجذري لا يُخفَّض/يُحذف.
